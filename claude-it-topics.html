Topic 01

IT Prompt Library

Prompt templates สำหรับงาน IT ที่ใช้บ่อยที่สุด — copy ไปปรับใช้ได้เลย

Prompt: Analyze Error / Log

วิเคราะห์ Error และ Log File

ใช้เมื่อต้องวิเคราะห์ error message หรือ log file เพื่อหา root cause

คุณเป็น Senior SysAdmin ที่เชี่ยวชาญ [Windows Server / Linux / Network] วิเคราะห์ error/log ที่แนบมา โดย: 1. ระบุ root cause ที่น่าจะเป็น (เรียงตาม probability) 2. อธิบายสาเหตุเป็นภาษาเข้าใจง่าย 3. ให้ step-by-step การแก้ไขพร้อม command จริง 4. บอก preventive measure เพื่อไม่ให้เกิดซ้ำ 5. ระบุว่า step ไหนที่ควรทำบน lab ก่อน ตอบเป็นภาษาไทย

Prompt: Write PowerShell Script

เขียน PowerShell Script พร้อมใช้งาน

ใช้เมื่อต้องการ script ที่มี error handling และ logging ครบถ้วน

คุณเป็น PowerShell expert เขียน script สำหรับ [Windows Server version] Task: [อธิบายสิ่งที่ต้องการทำ] Requirements: - Parameter input: [ระบุ parameters ที่ต้องการ] - Error handling: try/catch ครบถ้วน - Logging: บันทึก log พร้อม timestamp ไปที่ [path] - Output: แสดงผลลัพธ์แบบ verbose/summary - Comment: อธิบายทุก section เป็นภาษาไทย - Test: ระบุวิธีทดสอบ script บน lab ก่อน deploy

Prompt: Write KB Article

สร้าง Knowledge Base Article

ใช้สร้าง KB article สำหรับ end-user แก้ปัญหาเองได้

คุณเป็น IT Support ที่เขียน Knowledge Base สำหรับ end-user ทั่วไปของ TRAC เขียน KB article เรื่อง: [หัวข้อปัญหา] รูปแบบ: - ชื่อบทความ: เข้าใจง่าย ไม่เป็น technical - อาการที่พบ (bullet points) - สาเหตุที่พบบ่อย (ภาษาชาวบ้าน) - วิธีแก้ไขเองได้ (numbered steps ชัดเจน) - เมื่อไรควรโทรหา IT Support - Keywords สำหรับ search ภาษาไทย ระดับ non-technical user เข้าใจได้ทันที

Prompt: Draft Security Policy

ร่าง IT Security Policy

ใช้ร่าง policy ใหม่หรือ review policy เดิม

คุณเป็น IT Security expert และที่ปรึกษา compliance ร่าง [ชื่อ Policy] สำหรับ TRAC (บริษัทให้เช่ารถ พนักงาน 500+ คน) โปรด: 1. อ้างอิง PDPA พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล 2562 2. สอดคล้องกับ ISO 27001 framework (ถ้าเป็นไปได้) 3. ภาษาไทยเข้าใจง่าย ทั้งพนักงาน IT และ non-IT 4. Format: วัตถุประสงค์, ขอบเขต, นิยาม, หลักเกณฑ์, บทลงโทษ 5. ระบุส่วนที่ Legal/Management ต้อง review ก่อน

Prompt: Vendor Comparison

เปรียบเทียบ Vendor / Solution

ใช้สร้าง evaluation matrix สำหรับตัดสินใจเลือก solution

คุณเป็นที่ปรึกษา IT ที่ช่วยประเมิน solution สำหรับองค์กร เปรียบเทียบ: [Solution A] vs [Solution B] บริบท: TRAC บริษัทให้เช่ารถ IT team 5 คน budget จำกัด สร้าง evaluation matrix ที่รวม: 1. ราคา (TCO 3 ปี) 2. Features ที่ต้องการ vs nice-to-have 3. Integration กับ system ปัจจุบัน 4. Support และ SLA 5. Vendor stability / market position 6. คะแนนรวมพร้อม recommendation สรุปเป็นตาราง + paragraph สำหรับนำเสนอผู้บริหาร

Topic 02

Troubleshoot Framework

กระบวนการ troubleshoot ร่วมกับ Claude ที่ได้ผลดีที่สุด ใช้ได้กับปัญหาทุกประเภท

1

รวบรวมข้อมูลก่อนถาม

ก่อน paste ให้ Claude: ระบุ OS + version, error message เต็ม, สิ่งที่เปลี่ยนแปลงล่าสุด, สิ่งที่ลองทำไปแล้ว — ข้อมูลครบ = คำตอบดี

2

Paste Error + Log พร้อมกัน

อัพโหลด log file หรือ paste error message ทั้งหมด อย่าตัดทอน Claude ต้องการ full context เพื่อวิเคราะห์ได้ถูกต้อง

3

ขอ Prioritized Solutions

บอกให้ Claude เรียงลำดับวิธีแก้ตาม probability และ ความเร็ว — เริ่มจากวิธีง่ายที่สุดก่อน ไม่ต้องลอง advanced fix ตั้งแต่ต้น

4

ทดสอบ + Report ผล

ลองวิธีแรก แล้วกลับมาบอก Claude ว่าผลเป็นอย่างไร — ถ้าไม่สำเร็จ paste error ใหม่ ที่เกิดขึ้น Claude จะ debug ต่อได้ทันที

5

Document ทันทีหลังแก้สำเร็จ

หลังแก้ปัญหาได้ บอก Claude: "สรุป root cause และ solution เป็น KB article" — ใช้เวลา 2 นาที ประหยัดเวลาคราวหน้าได้มาก

Topic 03

Security Rules ที่ต้องรู้

กฎความปลอดภัยที่ IT ต้องปฏิบัติตามเมื่อใช้ Claude กับข้อมูลองค์กร

กฎเหล็ก #1

ห้ามใส่ password, API key, private key, token จริงเด็ดขาด ใช้ placeholder เช่น [PASSWORD], [API_KEY], [TOKEN] แทนเสมอ

กฎเหล็ก #2

ก่อนอัพโหลด log file ให้ anonymize ข้อมูลที่ sensitive ก่อน: IP จริง → 192.168.x.x, hostname จริง → SERVER-01, username จริง → USER-A

✅

Script Verification ก่อน Run เสมอ

ทุก script ที่ Claude เขียน ต้อง review code ก่อน รัน test บน lab environment ก่อน deploy บน production เสมอ ไม่มีข้อยกเว้น

📋

Change Management บน Production

Script หรือ config change ที่ Claude แนะนำ ถ้าจะ deploy บน production ต้องผ่าน Change Management process ของ TRAC ก่อนเสมอ

🔍

Double-check Security Policy

Policy ที่ Claude ร่างให้ ต้อง review โดย IT Manager หรือ Legal ก่อนประกาศใช้ Claude อาจไม่รู้ context เฉพาะของ TRAC

📊

Data ใน Dashboard = Anonymized

เมื่อสร้าง IT dashboard บน Cowork ข้อมูลที่ใช้ต้องไม่มี PII หรือข้อมูล confidential ของ TRAC ที่ไม่ควรเผยแพร่

Topic 04

Advanced Techniques สำหรับ IT

เทคนิคขั้นสูงที่จะทำให้ IT Admin ใช้ Claude ได้ในระดับที่เหนือกว่า

Technique 01

Multi-Step Script Building

แทนที่จะขอ script ซับซ้อนในครั้งเดียว ให้แบ่งเป็นขั้น: Step 1 ขอ pseudocode → Step 2 ขอ script พร้อม comment → Step 3 ขอเพิ่ม error handling → Step 4 ขอ test cases — ได้ script ที่สมบูรณ์กว่ามาก

Technique 02

Explain Before Execute

ก่อน run script ที่ Claude เขียน ให้ถาม: "อธิบายว่า script นี้ทำอะไรทีละบรรทัด มีความเสี่ยงอะไรบ้าง และมีอะไรที่ต้อง configure ก่อน run" — ช่วยให้เข้าใจและ review ได้ดีขึ้น

Technique 03

Iterative Documentation

สร้าง Runbook ได้เร็วโดย: ทำงาน IT ตามปกติ แล้ว"อธิบายสิ่งที่เพิ่งทำให้ Claude ฟัง" → Claude เขียน Runbook ให้ → review และแก้ไข — ดีกว่านั่งเขียนใหม่จาก 0

Technique 04

Cross-Platform Adaptation

ถ้าเจอ solution บน Windows ต้องการ Linux version บอก Claude: "แปลง PowerShell script นี้เป็น Bash script สำหรับ Ubuntu 22.04" — Claude ปรับ syntax และ command ให้ถูกต้องทันที

Technique 05

Preemptive Troubleshooting

ก่อนจะเกิดปัญหา ให้ถาม Claude: "ถ้าจะ migrate server นี้ ปัญหาอะไรที่มักเกิดขึ้น และจะ prevent ได้อย่างไร" — สร้าง checklist ป้องกันล่วงหน้าได้
